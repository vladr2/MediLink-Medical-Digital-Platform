\section{Arhitectura aplicațiilor web moderne}

\subsection{Modelul client-server și arhitectura REST}

Aplicațiile web moderne sunt construite pe baza modelului \textbf{client-server}, în care responsabilitățile sunt clar separate: clientul (browserul web sau aplicația mobilă) se ocupă de prezentarea datelor și interacțiunea cu utilizatorul, în timp ce serverul gestionează logica de business, persistența datelor și securitatea.

Comunicarea dintre client și server se realizează prin interfețe de tip \textbf{API} (\textit{Application Programming Interface}). Stilul arhitectural dominant în aplicațiile web actuale este \textbf{REST} (\textit{Representational State Transfer}), definit de Roy Fielding în teza sa de doctorat din 2000 \cite{fielding2000rest}. Un API RESTful organizează resursele aplicației ca entități adresabile prin URI-uri (\textit{Uniform Resource Identifiers}), manipulate prin metodele standard ale protocolului HTTP.

Principiile fundamentale ale REST includ:

\begin{itemize}
  \item \textbf{Interfață uniformă} --- resurse identificate prin URI-uri distincte, manipulate exclusiv prin metodele standard HTTP: GET (citire), POST (creare), PUT/PATCH (actualizare), DELETE (ștergere);
  \item \textbf{Statelessness} --- fiecare cerere HTTP conține toate informațiile necesare procesării sale, serverul nu păstrează starea sesiunii între cereri consecutive; autentificarea se realizează prin token-uri incluse în fiecare cerere;
  \item \textbf{Cacheability} --- răspunsurile pot fi marcate explicit ca putând fi memorate în cache (prin header-ele \code{Cache-Control}, \code{ETag}), reducând sarcina serverului;
  \item \textbf{Sistem pe niveluri} --- clientul nu trebuie să știe dacă comunică direct cu serverul final sau prin intermediari (load balancer, proxy invers, CDN);
  \item \textbf{Reprezentare a resurselor} --- aceeași resursă poate fi livrată în formate diferite (JSON, XML, HTML) pe baza header-ului \code{Accept} din cerere.
\end{itemize}

MediLink implementează un API RESTful complet, cu peste 80 de endpoint-uri organizate logic pe resurse distincte: \code{/api/auth}, \code{/api/users}, \code{/api/patients}, \code{/api/doctors}, \code{/api/appointments}, \code{/api/medical-records}, \code{/api/vitals}, \code{/api/messages}, \code{/api/notifications} și altele. Toate răspunsurile API utilizează formatul JSON, cu o structură consistentă care include câmpurile de date și, în caz de eroare, un mesaj descriptiv și un cod de eroare.

\subsection{Arhitectura Single Page Application (SPA)}

Frontendurile moderne sunt construite ca \textbf{Single Page Applications (SPA)}, în care navigarea între secțiuni nu implică reîncărcarea completă a paginii din server. Browserul descarcă inițial un pachet de resurse JavaScript, CSS și HTML, după care toate tranzițiile de navigare sunt gestionate de client prin actualizarea dinamică a DOM-ului (\textit{Document Object Model}).

Comunicarea cu serverul se face exclusiv prin apeluri API asincrone (fetch/XHR/HttpClient), fără reîncărcare de pagină. Această arhitectură aduce o experiență de utilizare fluidă, similară aplicațiilor native desktop sau mobile, și o separare clară a responsabilităților: frontendului îi revine exclusiv prezentarea și logica de interacțiune cu utilizatorul, în timp ce backendului îi revine logica de business și persistența datelor.

\subsection{Arhitectura pe trei niveluri}

Aplicațiile web enterprise moderne adoptă o \textbf{arhitectură pe trei niveluri} (\textit{three-tier architecture}):

\begin{enumerate}
  \item \textbf{Nivelul de prezentare} (\textit{Presentation Tier}) --- interfața grafică cu utilizatorul, implementată ca SPA Angular în cazul MediLink; responsabilă exclusiv de afișare și captarea inputului utilizatorului;
  \item \textbf{Nivelul logic} (\textit{Application/Business Tier}) --- serverul de aplicație FastAPI, care implementează regulile de business, autentificarea, autorizarea și orchestrează accesul la date;
  \item \textbf{Nivelul de date} (\textit{Data Tier}) --- baza de date PostgreSQL și cache-ul Redis, responsabile de persistența și recuperarea eficientă a datelor.
\end{enumerate}

Această separare permite scalabilitate independentă pe fiecare nivel, înlocuirea unui nivel fără impact asupra celorlalte și o testabilitate mai bună a componentelor.

\section{FastAPI --- Framework-ul backend}

\subsection{Prezentare generală}

\textbf{FastAPI} \cite{fastapi2024} este un framework web modern pentru Python, creat de Sebastián Ramírez și lansat în 2018. Bazat pe standardul \textbf{ASGI} (\textit{Asynchronous Server Gateway Interface}), FastAPI suportă nativ programarea asincronă prin cuvintele cheie \code{async}/\code{await} ale lui Python 3.7+, permițând gestionarea unui număr mare de conexiuni concurente cu un consum redus de resurse.

Spre deosebire de WSGI (\textit{Web Server Gateway Interface}) --- standardul anterior, utilizat de Django și Flask --- ASGI este proiectat pentru a gestiona conexiuni de lungă durată (WebSockets, Server-Sent Events) și operații I/O concurente fără a bloca thread-ul principal. Aceasta este o proprietate esențială pentru MediLink, care utilizează conexiuni WebSocket persistente pentru notificări și teleconsultații.

Conform benchmark-urilor independente realizate de TechEmpower \cite{techempower2024}, FastAPI se situează printre cele mai performante framework-uri web Python, cu viteze comparabile cu framework-uri Go și Node.js în scenariile de I/O-bound (acces la baze de date, apeluri HTTP externe).

\subsection{Programare asincronă cu asyncio}

Fundația performanței FastAPI este biblioteca \textbf{asyncio} \cite{python_asyncio2024} din biblioteca standard Python, care implementează un model de concurență bazat pe \textit{event loop}. Spre deosebire de threading-ul tradițional (în care fiecare conexiune ocupă un thread dedicat cu costul asociat de memorie și context switching), modelul asyncio utilizează un singur thread care alternează eficient între multiple operații de I/O în așteptare.

Când o funcție asincronă FastAPI execută o operație de I/O (interogare bază de date, apel API extern), controlul este cedat event loop-ului, care poate procesa alte cereri în același timp. Rezultatul este un server capabil să gestioneze mii de conexiuni concurente cu un singur proces Python, cu consum de memorie constant.

\subsection{Validare cu Pydantic}

\textbf{Pydantic} \cite{pydantic2024} este biblioteca de validare a datelor integrată în FastAPI. Schema datelor se definește o singură dată ca modele Python cu tipuri adnotate, iar FastAPI generează automat logica de validare, serializare și documentare OpenAPI:

\begin{lstlisting}[language=Python3, caption={Modele Pydantic pentru gestionarea utilizatorilor}]
from pydantic import BaseModel, EmailStr, field_validator
from typing import Optional
from datetime import date

class UserCreate(BaseModel):
    email: EmailStr
    password: str
    first_name: str
    last_name: str
    role: str

    @field_validator('password')
    @classmethod
    def password_strength(cls, v):
        if len(v) < 8:
            raise ValueError('Parola trebuie sa aiba minim 8 caractere')
        return v

class UserResponse(BaseModel):
    id: str
    email: str
    first_name: Optional[str]
    last_name: Optional[str]
    role: str
    is_active: bool

    model_config = {"from_attributes": True}
\end{lstlisting}

Orice cerere HTTP cu date invalide (email incorect, câmp lipsă, tip greșit) primește automat un răspuns HTTP 422 cu detalii precise despre câmpurile eronate, fără nicio linie de cod adițional.

\subsection{Documentație automată OpenAPI}

FastAPI generează automat documentație interactivă bazată pe specificația \textbf{OpenAPI 3.0}: o interfață Swagger UI accesibilă la \code{/api/docs} și o documentație ReDoc la \code{/api/redoc}. Aceasta permite testarea completă a API-ului direct din browser, vizualizarea schemelor de request/response și exportul specificației în format JSON pentru integrări cu alte instrumente.

\subsection{Dependency Injection}

Sistemul de \textbf{dependency injection} al FastAPI permite declararea și injectarea automată a resurselor comune (sesiunea bazei de date, utilizatorul autentificat, permisiunile RBAC) în funcțiile handler, eliminând codul duplicat:

\begin{lstlisting}[language=Python3, caption={Dependency Injection pentru autentificare și baza de date}]
from fastapi import Depends, HTTPException
from app.middleware.auth import get_current_user
from app.core.database import get_db

@router.get("/medical-records/patient/{patient_id}")
async def get_patient_records(
    patient_id: str,
    current_user = Depends(get_current_user),
    db: Session = Depends(get_db),
):
    if current_user.role not in ["doctor", "admin", "assistant"]:
        raise HTTPException(403, "Acces interzis")
    # logica de business
\end{lstlisting}

\section{TypeScript și Angular 19 --- Framework-ul frontend}

\subsection{TypeScript --- JavaScript cu tipizare statică}

\textbf{TypeScript} \cite{typescript2024} este un superset al JavaScript-ului, dezvoltat de Microsoft, care adaugă tipizare statică opțională și alte caracteristici moderne. Codul TypeScript este transpilat în JavaScript standard înainte de execuție, deci rulează în orice browser sau mediu Node.js fără modificări.

Principalele avantaje ale TypeScript în contextul MediLink:

\begin{itemize}
  \item \textbf{Detectarea erorilor la compilare} --- greșelile de tip (accesarea unui câmp inexistent, transmiterea unui parametru de tip greșit) sunt detectate de compilator, nu la rulare în producție;
  \item \textbf{Autocompletare și refactorizare} --- IDE-urile (VSCode, WebStorm) oferă sugestii precise și refactorizare sigură bazate pe informațiile de tip;
  \item \textbf{Documentație implicită} --- interfețele TypeScript servesc ca documentație live a structurii datelor, sincronizată automat cu codul;
  \item \textbf{Scalabilitate} --- într-o bază de cod mare (MediLink frontend are 80+ componente și servicii), tipizarea previne categorii întregi de bug-uri de integrare.
\end{itemize}

\subsection{Angular 19 --- Prezentare generală}

\textbf{Angular} \cite{angular2024} este un framework TypeScript complet pentru construirea aplicațiilor web, dezvoltat și menținut de Google. Spre deosebire de biblioteci UI precum React sau Vue, Angular este un framework \textit{opinionated} care include soluții integrate pentru rutare, formulare reactive, comunicare HTTP, testare unitară și injecția de dependențe.

MediLink utilizează \textbf{Angular 19}, care aduce îmbunătățiri semnificative față de versiunile anterioare: componente standalone (fără module NgModule), Ivy renderer optimizat pentru bundle size redus, suport nativ pentru semnale reactive și control flow sintactic integrat în template-uri (\code{@if}, \code{@for}).

\subsection{Componente Standalone}

Arhitectura MediLink adoptă în întregime paradigma \textbf{componentelor standalone}, stabilizată în Angular 17. Fiecare componentă declară explicit dependențele sale, eliminând necesitatea modulelor NgModule și simplificând structura aplicației:

\subsection{Programare reactivă cu RxJS}

\textbf{RxJS} \cite{rxjs2024} (\textit{Reactive Extensions for JavaScript}) este biblioteca de programare reactivă integrată în Angular, bazată pe conceptul de \textbf{Observables} --- fluxuri de date asincrone care pot fi transformate, combinate și subscrise.

MediLink utilizează RxJS pentru gestionarea tuturor operațiilor asincrone:

\begin{itemize}
  \item \textbf{HttpClient} returnează \code{Observable<T>} pentru fiecare cerere HTTP, permițând anularea automată la distrugerea componentei (prevenirea memory leaks);
  \item \textbf{Subject} și \textbf{BehaviorSubject} sunt utilizate pentru comunicarea între componente și servicii (starea utilizatorului autentificat, notificările WebSocket);
  \item \textbf{Operatori pipe} (\code{map}, \code{tap}, \code{catchError}, \code{debounceTime}) permit transformarea și filtrarea fluxurilor de date fără side effects.
\end{itemize}

\section{PostgreSQL --- Baza de date relațională}

\subsection{Prezentare generală}

\textbf{PostgreSQL} \cite{postgresql2024} este un sistem de gestiune a bazelor de date relaționale (\textit{RDBMS}) open-source cu o istorie de peste 35 de ani de dezvoltare activă. Recunoscut pentru conformitatea strictă cu standardele SQL, fiabilitate în medii critice și un set bogat de tipuri de date avansate, PostgreSQL este soluția preferată pentru aplicații care necesită integritate ridicată a datelor.

MediLink utilizează PostgreSQL 16 și beneficiază de mai multe caracteristici avansate:

\begin{itemize}
  \item \textbf{UUID nativ} --- tipul \code{uuid} este utilizat ca cheie primară pentru toate entitățile, generată prin funcția \code{gen\_random\_uuid()};
  \item \textbf{JSONB} --- tipul binar JSON permite stocarea datelor semi-structurate (lista medicamentelor dintr-o prescripție, mesajele unui chat) cu indexare și interogare eficientă;
  \item \textbf{Tipuri ENUM} --- valorile predefinite (roluri utilizator, statusuri programare) sunt stocate ca tipuri enumerate PostgreSQL, garantând integritatea la nivel de bază de date;
  \item \textbf{Tranzacții ACID} --- atomicitate, consistență, izolare și durabilitate garantate pentru operațiile complexe multi-tabel;
  \item \textbf{Indexuri parțiale} --- indexuri create doar pentru subseturi de date (ex: programările viitoare), reducând dimensiunea indexului și îmbunătățind performanța.
\end{itemize}

\subsection{Normalizarea bazei de date}

Schema MediLink respectă \textbf{forma normală trei} (3NF), eliminând redundanța datelor și anomaliile de actualizare. Separarea datelor de autentificare (tabelul \code{users}) de profilul medical (tabelul \code{patients}) și de profilul profesional (tabelul \code{doctors}) permite extinderea independentă a fiecărei entități și acordarea de permisiuni granulare la nivel de tabel.

Relațiile many-to-many (medic-pacient, utilizator-rol) sunt implementate prin tabele de legătură (\code{doctor\_patient}), conform principiilor modelării relaționale.

\subsection{SQLAlchemy ORM}

\textbf{SQLAlchemy} \cite{sqlalchemy2024} este cel mai utilizat ORM (\textit{Object-Relational Mapper}) pentru Python, oferind o abstractizare completă a bazei de date prin maparea tabelelor SQL la clase Python. MediLink utilizează SQLAlchemy 2.0 cu stilul declarativ modern:

\begin{lstlisting}[language=Python3, caption={Model SQLAlchemy pentru entitatea User}]
class User(Base):
    __tablename__ = "users"

    id             = Column(UUID(as_uuid=True), primary_key=True,
                            default=uuid.uuid4)
    email          = Column(String, unique=True, nullable=False, index=True)
    first_name     = Column(String, nullable=True)
    last_name      = Column(String, nullable=True)
    hashed_password= Column(String, nullable=True)
    role           = Column(Enum(UserRole), nullable=False)
    phone          = Column(String, nullable=True)
    birth_date     = Column(String, nullable=True)
    mfa_secret     = Column(String, nullable=True)
    is_active      = Column(Boolean, default=True)
    created_at     = Column(DateTime(timezone=True),
                            server_default=func.now())

    # Relatii
    doctor_profile  = relationship("Doctor", back_populates="user",
                                   uselist=False)
    patient_profile = relationship("Patient", back_populates="user",
                                   uselist=False)
    notifications   = relationship("Notification", back_populates="user")
\end{lstlisting}

\subsection{Alembic --- Migrații ale schemei}

Gestionarea evoluției schemei bazei de date se realizează prin \textbf{Alembic} \cite{alembic2024}. Fiecare modificare structurală este reprezentată ca un fișier Python versionat, cu funcții \code{upgrade()} și \code{downgrade()}, formând un lanț de versiuni trasat prin câmpurile \code{revision} și \code{down\_revision}. Acest mecanism permite aplicarea și revenirea modificărilor în mod controlat atât în mediul de dezvoltare, cât și în producție.

\section{Redis --- Cache și date în memorie}

\textbf{Redis} \cite{redis2024} (\textit{Remote Dictionary Server}) este o bază de date în memorie de tip cheie-valoare, cu timp de acces sub milisecundă. MediLink utilizează Redis pentru trei scenarii distincte:

\begin{itemize}
  \item \textbf{Blacklist refresh token-uri} --- la logout, refresh token-ul este invalidat prin adăugarea sa în Redis cu TTL egal cu durata de viață rămasă a token-ului; la fiecare cerere de reînnoire, serverul verifică prezența token-ului în blacklist înainte de a emite un nou access token;
  \item \textbf{Registrul conexiunilor WebSocket} --- maparea dintre ID-ul utilizatorului și conexiunea WebSocket activă este stocată în Redis, permitând trimiterea mesajelor direcționate chiar și în arhitecturi cu mai multe instanțe backend;
  \item \textbf{Rate limiting} --- contoarele pentru limitarea numărului de cereri per IP sunt gestionate atomic în Redis prin comenzile \code{INCR} și \code{EXPIRE}.
\end{itemize}

\section{Securitate}

\subsection{Autentificare JWT}

\textbf{JSON Web Token} (JWT) \cite{rfc7519} este un standard deschis (RFC 7519) pentru transmiterea securizată a informațiilor între părți ca obiect JSON semnat criptografic. Un JWT este compus din trei segmente codificate Base64URL, separate prin punct: \textbf{header} (algoritmul de semnare), \textbf{payload} (informațiile transportate --- claims) și \textbf{semnătura} digitală.

MediLink implementează un flux de autentificare cu două token-uri:

\begin{itemize}
  \item \textbf{Access token} --- cu valabilitate de 15 minute, transportat în header-ul HTTP \code{Authorization: Bearer <token>}, conține ID-ul utilizatorului și rolul acestuia; durata scurtă limitează fereastra de expunere în cazul interceptării;
  \item \textbf{Refresh token} --- cu valabilitate de 7 zile, stocat în cookie HTTP-Only (inaccesibil din JavaScript, protejat împotriva atacurilor XSS), utilizat exclusiv pentru obținerea unor noi access token-uri la expirarea acestora.
\end{itemize}

Algoritmul de semnare utilizat este \textbf{HS256} (HMAC-SHA256), cu o cheie secretă de minimum 256 de biți generată aleatoriu la instalare și stocată în variabile de mediu.

\subsection{Autentificare în doi factori (2FA/TOTP)}

MediLink implementează autentificarea în doi factori opțională bazată pe protocolul \textbf{TOTP} (\textit{Time-based One-Time Password}), standardizat în RFC 6238 \cite{rfc6238}. Algoritmul generează coduri numerice de 6 cifre cu valabilitate de 30 de secunde, pe baza unui secret partajat și a timpului curent al sistemului. Implementarea este compatibilă cu aplicațiile Google Authenticator, Authy și Microsoft Authenticator.

Fluxul de activare 2FA este:
\begin{enumerate}
  \item Serverul generează un secret TOTP unic pentru utilizator și îl criptează cu Fernet;
  \item Se afișează utilizatorului un cod QR scanabil (codificat conform standardului \textit{Key URI Format}) și cheia secretă în format text;
  \item Utilizatorul scanează codul QR cu aplicația de autentificare și introduce un cod valid pentru confirmare;
  \item Secretul criptat este stocat în câmpul \code{mfa\_secret} al utilizatorului.
\end{enumerate}

\subsection{Criptarea datelor medicale sensibile}

Datele medicale sensibile sunt criptate la nivel de coloană în baza de date folosind \textbf{Fernet} \cite{cryptography2024} --- o implementare simetrică autentificată bazată pe AES-128-CBC cu HMAC-SHA256. Fernet garantează că datele criptate nu pot fi alterate sau decriptate fără cheia corectă.

Criptarea este transparentă pentru codul de business, implementată printr-un \textbf{TypeDecorator SQLAlchemy} personalizat care interceptează automat operațiile de citire și scriere la nivel de coloană. Cheia de criptare este stocată exclusiv ca variabilă de mediu, niciodată în codul sursă sau în baza de date.

\subsection{RBAC --- Control al accesului bazat pe roluri}

\textbf{Role-Based Access Control} (RBAC) \cite{sandhu1996rbac} este un model de control al accesului în care permisiunile sunt asociate rolurilor, nu utilizatorilor individuali. Această abordare simplifică administrarea într-un sistem cu mulți utilizatori: modificarea permisiunilor unui rol afectează automat toți utilizatorii care dețin acel rol.

MediLink definește patru roluri cu seturi distincte de permisiuni: \code{admin} (acces complet la toate resursele și funcționalitățile administrative), \code{doctor} (acces la pacienții proprii, înregistrări medicale, programări și funcționalitățile AI), \code{assistant} (gestionarea programărilor, înregistrarea semnelor vitale, atribuirea pacienților) și \code{patient} (acces la propriile date medicale, programări și comunicare cu medicul).

\subsection{Securitate conform OWASP}

Proiectarea securității MediLink a luat în considerare \textbf{OWASP Top 10} \cite{owasp2021}, lista celor mai critice vulnerabilități ale aplicațiilor web. Principalele măsuri implementate:

\begin{itemize}
  \item \textbf{Injection} (A03) --- parametrii SQL sunt transmisi exclusiv prin query parameters SQLAlchemy, elimând complet riscul SQL injection;
  \item \textbf{Broken Authentication} (A07) --- bcrypt cu factor de cost 12 pentru hashing parole, JWT cu durată scurtă, 2FA opțional;
  \item \textbf{Sensitive Data Exposure} (A02) --- criptare Fernet pentru datele medicale, cookie-uri HTTP-Only, HTTPS obligatoriu în producție;
  \item \textbf{Broken Access Control} (A01) --- RBAC granular verificat pe server pentru fiecare endpoint, utilizatorii nu pot accesa datele altor utilizatori;
  \item \textbf{Security Misconfiguration} (A05) --- CORS configurat restrictiv, header-e de securitate (X-Content-Type-Options, X-Frame-Options) adăugate prin Nginx.
\end{itemize}

\section{Inteligența artificială în medicină}

\subsection{Large Language Models (LLM)}

\textbf{Modelele lingvistice mari} (LLM) sunt rețele neuronale de tip transformator \cite{vaswani2017attention} antrenate pe corpusuri masive de text, capabile să genereze text coerent, să răspundă la întrebări complexe și să realizeze raționamente în lanț. Arhitectura \textit{transformer}, introdusă în 2017, utilizează mecanismul de \textbf{atenție} (\textit{self-attention}) pentru a captura dependențele între tokeni dintr-o secvență, indiferent de distanța dintre ei.

Studii recente au demonstrat că modelele LLM de mari dimensiuni pot codifica cunoștințe clinice de nivel expert: în testul \textit{United States Medical Licensing Examination} (USMLE), modelul Med-PaLM 2 a atins performanțe de nivel expert, depășind pragul de promovare cu o marjă semnificativă \cite{singhal2023large}.

\subsection{Modelul Llama 3.3 70B și API-ul Groq}

Modelul \textbf{Llama 3.3 70B} \cite{meta2024llama}, dezvoltat de Meta AI, este un model open-weights cu 70 de miliarde de parametri, optimizat pentru instrucțiuni și conversație. MediLink utilizează acest model prin \textbf{API-ul Groq} \cite{groq2024}, care oferă inferență LLM extrem de rapidă (150+ token-uri/secundă) prin hardware specializat LPU (\textit{Language Processing Unit}).

Avantajele acestei combinații pentru MediLink:
\begin{itemize}
  \item Latență redusă (răspuns în sub 2 secunde pentru prompturi medicale de lungime medie);
  \item Model open-source --- nu există risc de vendor lock-in; dacă API-ul Groq devine indisponibil, modelul poate fi rulat local;
  \item Calitate ridicată a răspunsurilor în limba română, relevantă pentru utilizatorii platformei.
\end{itemize}

\subsection{Aplicații AI în MediLink}

\textbf{Chatbot medical contextual} --- pacienții pot conversa cu un asistent AI medical care menține contextul conversației (ultimele 10 schimburi), are acces la profilul medical al pacientului și este instruit printr-un prompt de sistem să ofere informații generale bazate pe dovezi, să recomande consultul medical pentru situații urgente și să nu substituie sfatul medicului.

\textbf{Predicție risc clinic} --- pentru fiecare pacient, medicul poate solicita o evaluare AI a riscului medical pe o scală de la 1 la 10. Modelul primește ca context: vârsta și sexul pacientului, condițiile cronice, medicamentele prescrise, ultimele analize și semnele vitale recente. Răspunsul structurat include scorul de risc, nivelul de risc (scăzut/mediu/ridicat/critic), factorii identificați și recomandări clinice concrete.

\textbf{Generator de raport PDF} --- la cererea medicului, AI-ul generează un raport medical narativ complet pentru un pacient, structurat pe secțiuni clinice (sumar, evoluție, plan terapeutic, recomandări). Raportul generat este convertit în format PDF prin biblioteca WeasyPrint \cite{weasyprint2024}.

\section{Comunicare în timp real}

\subsection{WebSockets --- Protocol și utilizare}

\textbf{WebSockets} \cite{rfc6455} (RFC 6455) reprezintă un protocol de comunicare bidirecțional full-duplex peste o singură conexiune TCP persistentă. Procesul de stabilire a conexiunii WebSocket pornește cu un \textit{HTTP Upgrade handshake}: clientul trimite o cerere HTTP cu header-ele \code{Upgrade: websocket} și \code{Connection: Upgrade}, iar serverul răspunde cu codul 101 (Switching Protocols).

Spre deosebire de HTTP polling (interogare periodică a serverului) sau long-polling, WebSockets elimină overhead-ul HTTP per mesaj și permit serverului să inițieze transmiterea datelor fără o cerere anterioară din partea clientului. Aceasta este esențial pentru notificările în timp real din MediLink, unde latența redusă și eficiența rețelei sunt critice.

\subsection{WebRTC --- Teleconsultații video}

\textbf{WebRTC} (\textit{Web Real-Time Communication}) \cite{webrtc2024} este un set de standarde și API-uri web care permit comunicarea audio-video și schimbul de date direct între browsere (\textit{peer-to-peer}), fără a necesita un server intermediar pentru fluxul media.

Procesul de stabilire a unei sesiuni WebRTC implică:
\begin{enumerate}
  \item \textbf{Semnalizare} --- schimbul de descriptori de sesiune SDP (\textit{Session Description Protocol}) și candidați ICE (\textit{Interactive Connectivity Establishment}) prin canalul WebSocket al MediLink;
  \item \textbf{Negocierea codecurilor} --- browserele negociază codecurile audio (Opus) și video (VP8/H.264) suportate de ambele părți;
  \item \textbf{Traversarea NAT} --- protocolul ICE determină calea optimă de conectivitate, utilizând servere STUN (\textit{Session Traversal Utilities for NAT}) pentru descoperirea adresei publice;
  \item \textbf{Transmiterea securizată} --- fluxurile audio/video sunt criptate cu DTLS-SRTP și transmise direct între browsere, fără a trece prin serverul MediLink.
\end{enumerate}

Avantajul major al abordării peer-to-peer este că serverul MediLink nu procesează fluxurile media, eliminând costurile de bandwidth și latența adițională unui server relay.

\section{Containerizare cu Docker}

\textbf{Docker} \cite{docker2024} este o platformă de containerizare care împachetează aplicațiile împreună cu toate dependențele lor (biblioteci, configurații, variabile de mediu) într-o unitate izolată portabilă numită \textbf{container}. Spre deosebire de mașinile virtuale, containerele partajează kernel-ul sistemului de operare gazdă, oferind izolare completă cu overhead minim.

\textbf{Docker Compose} permite definirea și orchestrarea unui stack multi-container printr-un singur fișier YAML. Fiecare serviciu definit în \code{docker-compose.yml} specifică imaginea Docker, variabilele de mediu, porturile expuse, volumele pentru persistența datelor și dependențele față de alte servicii.

Containerizarea aduce beneficii practice concrete pentru MediLink:
\begin{itemize}
  \item \textbf{Portabilitate} --- întreaga platformă rulează identic pe orice sistem cu Docker instalat (Windows, Linux, macOS), eliminând problemele clasice de tip \textit{works on my machine};
  \item \textbf{Reproducibilitate} --- versiunile exacte ale tuturor dependențelor sunt fixate în Dockerfile, garantând că mediul de producție este identic cu cel de dezvoltare;
  \item \textbf{Izolare} --- fiecare serviciu rulează în propriul container cu propriul sistem de fișiere, prevenind conflictele între dependențe;
  \item \textbf{Deployment simplificat} --- întreaga platformă MediLink pornește cu o singură comandă: \code{docker compose up --build}.
\end{itemize}

\section{Nginx --- Proxy invers și server web}

\textbf{Nginx} \cite{nginx2024} este un server web de înaltă performanță utilizat în MediLink în rolul de \textbf{proxy invers} (\textit{reverse proxy}): primește toate cererile HTTP/HTTPS din exterior și le rutează către serviciile backend corespunzătoare, în funcție de URI și tipul conexiunii.

Rolurile Nginx în arhitectura MediLink:
\begin{itemize}
  \item \textbf{Servire fișiere statice} --- fișierele Angular compilate (JavaScript, CSS, imagini) sunt servite direct de Nginx, fără a implica serverul FastAPI;
  \item \textbf{Proxy pentru API} --- cererile cu prefix \code{/api} sunt redirecționate către serviciul backend FastAPI;
  \item \textbf{Proxy WebSocket} --- cererile de upgrade WebSocket sunt gestionate corect prin configurarea header-elor \code{Upgrade} și \code{Connection};
  \item \textbf{Rate limiting} --- limite de cereri per IP configurate pentru protecția endpoint-urilor sensibile;
  \item \textbf{Compresie Gzip} --- răspunsurile text (JSON, HTML, CSS, JavaScript) sunt comprimate automat, reducând bandwidth-ul cu 60-80\%.
\end{itemize}

\section{GDPR și protecția datelor medicale}

\textbf{Regulamentul General privind Protecția Datelor} (GDPR) \cite{gdpr2016} (Regulamentul UE 2016/679) stabilește cadrul legal pentru procesarea datelor personale în Uniunea Europeană. Datele medicale sunt clasificate ca \textbf{date speciale} (articolul 9 GDPR), care beneficiază de protecție sporită și pot fi procesate numai în condiții strict definite.

Principalele drepturi ale persoanei vizate relevante pentru MediLink:
\begin{itemize}
  \item \textbf{Dreptul de acces} (Art. 15) --- operatorul trebuie să furnizeze o copie a datelor personale prelucrate la cererea persoanei vizate;
  \item \textbf{Dreptul la portabilitatea datelor} (Art. 20) --- datele trebuie furnizate într-un format structurat, utilizat frecvent și lizibil automat;
  \item \textbf{Dreptul la rectificare} (Art. 16) --- datele incorecte sau incomplete trebuie corectate fără întârzieri nejustificate;
  \item \textbf{Principiul minimizării datelor} (Art. 5.1.c) --- se colectează și se prelucrează exclusiv datele necesare scopului declarat.
\end{itemize}

MediLink implementează toate aceste cerințe prin funcționalități tehnice dedicate, descrise în detaliu în Capitolul 3.
