\section{Arhitectura generală a sistemului}

MediLink este construit pe o arhitectură modernă cu trei niveluri (\textit{three-tier architecture}): prezentare (frontend Angular), logică de business (backend FastAPI) și persistență (PostgreSQL + Redis). Figura \ref{fig:arhitectura} ilustrează componentele principale și fluxul de comunicare dintre ele.

\begin{figure}[H]
  \centering
  \caption{Arhitectura generală a platformei MediLink}
  \label{fig:arhitectura}
\end{figure}

Toate serviciile rulează ca containere Docker orchestrate prin Docker Compose, cu Nginx ca reverse proxy care rutează cererile HTTP/HTTPS și conexiunile WebSocket către serviciile corespunzătoare. Această arhitectură garantează că întreaga platformă poate fi pornită cu o singură comandă (\code{docker compose up --build}), indiferent de sistemul de operare gazdă.

Separarea clară a responsabilităților permite scalabilitate independentă: în cazul unei creșteri a numărului de utilizatori, se pot lansa instanțe multiple de backend (FastAPI suportă nativ concurența asincronă) fără modificări ale frontend-ului sau bazei de date.

\subsection{Cheile primare UUID}

O decizie de proiectare importantă a fost utilizarea \textbf{UUID v4} ca cheie primară pentru toate entitățile, în locul ID-urilor întregi auto-incrementate convenționale. Avantajele acestei abordări în contextul medical sunt:
\begin{itemize}
  \item \textbf{Securitate} — ID-urile UUID nu sunt predictibile, prevenind enumerarea resurselor (\textit{IDOR attacks});
  \item \textbf{Scalabilitate} — unicitatea globală permite generarea cheilor pe client sau în sisteme distribuite;
  \item \textbf{Integritate referențială} — mai ușor de verificat în auditare față de întregi.
\end{itemize}

\subsection{Criptarea transparentă — EncryptedString}

O contribuție tehnică originală este implementarea unui \textbf{TypeDecorator SQLAlchemy} personalizat pentru criptarea transparentă a câmpurilor sensibile:

\begin{lstlisting}[language=Python3, caption={TypeDecorator pentru criptare transparentă}]
from sqlalchemy.types import TypeDecorator, String
from cryptography.fernet import Fernet
import os

class EncryptedString(TypeDecorator):
    impl = String
    cache_ok = True

    def __init__(self):
        super().__init__()
        key = os.environ.get("ENCRYPTION_KEY", "")
        self._fernet = Fernet(key.encode()) if key else None

    def process_bind_param(self, value, dialect):
        """Criptare la scriere în DB"""
        if value is None or self._fernet is None:
            return value
        return self._fernet.encrypt(value.encode()).decode()

    def process_result_value(self, value, dialect):
        """Decriptare la citire din DB"""
        if value is None or self._fernet is None:
            return value
        try:
            return self._fernet.decrypt(value.encode()).decode()
        except Exception:
            return value
\end{lstlisting}

Prin utilizarea \code{EncryptedString} ca tip de coloană SQLAlchemy, câmpurile sensibile (diagnostic, tratament, rezultate analize, CNP) sunt criptate automat la orice scriere în baza de date și decriptate transparent la orice citire, fără ca restul codului de business să fie conștient de acest mecanism.

\subsection{Rate Limiting}

Pentru protecția împotriva atacurilor de tip brute-force și DoS (\textit{Denial of Service}), MediLink implementează \textbf{rate limiting} la două niveluri:

\begin{itemize}
  \item \textbf{Nginx} — limită de 5 cereri/minut pentru endpoint-ul de login, 60 cereri/minut pentru API-ul general;
  \item \textbf{Aplicație} — biblioteca \code{slowapi} aplică limitări granulare per endpoint, cu identificare bazată pe IP.
\end{itemize}

\section{Sistemul de autentificare și securitate}

\subsection{Fluxul de autentificare}

Autentificarea în MediLink urmează un flux securizat în mai mulți pași, ilustrat în figura \ref{fig:auth_flow}:

\begin{enumerate}
  \item Utilizatorul trimite credențialele (email + parolă) prin POST la \code{/api/auth/login};
  \item Serverul verifică parola folosind \textbf{bcrypt} cu factor de cost 12;
  \item Dacă 2FA este activat, serverul returnează un token temporar și cere codul TOTP;
  \item La verificarea cu succes, serverul emite un access token JWT (15 min) și un refresh token HTTP-Only (7 zile);
  \item La expirarea access token-ului, clientul apelează automat \code{/api/auth/refresh} pentru reînnoire, transparent pentru utilizator.
\end{enumerate}

\subsection{Middleware-ul de autentificare}

Verificarea token-ului JWT se realizează printr-un middleware FastAPI (\code{get\_current\_user}) injectat ca dependență în toate endpoint-urile protejate. Middleware-ul extrage token-ul din header-ul \code{Authorization}, îl verifică criptografic, verifică expirarea și returnează obiectul utilizator din baza de date.

\subsection{Middleware RBAC}

Controlul accesului granular este implementat prin funcții de dependență FastAPI specializate per rol. Fiecare endpoint declară explicit ce roluri au permisiunea de acces, iar verificarea se efectuează automat la fiecare cerere, înainte ca logica de business să fie executată. Utilizatorii cu roluri nepotrivite primesc răspunsul HTTP 403 Forbidden, fără ca cererea să ajungă vreodată la logica de prelucrare a datelor.

\section{Modulul de fișe medicale}

\subsection{Tipologia înregistrărilor medicale}

Fișa medicală electronică în MediLink este structurată în cinci tipuri de înregistrări, fiecare cu câmpuri specifice:

\begin{itemize}
  \item \textbf{Consultație} (\code{consultatie}) — diagnostic, tratament, observații clinice;
  \item \textbf{Analiză de laborator} (\code{analiza}) — rezultatele analizelor, valori de referință, interpretare;
  \item \textbf{Tratament} (\code{tratament}) — schema terapeutică completă cu doze și durată;
  \item \textbf{Investigație paraclinică} (\code{investigatie}) — rezultate imagistice, EKG, spirometrie etc.;
  \item \textbf{Rețetă} (\code{reteta}) — lista medicamentelor prescrise cu doze și instrucțiuni.
\end{itemize}

\subsection{Detectarea automată a anomaliilor}

Fiecare înregistrare medicală poate fi marcată cu indicatorul \code{has\_anomaly} și o notă descriptivă \code{anomaly\_notes}. Această funcționalitate permite medicilor să evidențieze rapid înregistrările care necesită atenție (valori anormale de laborator, evoluție negativă) și pacienților să fie alertați prompt.

\subsection{Paginarea și filtrarea}

Lista de fișe medicale din interfața medicului implementează \textbf{paginare manuală} pe o grilă de carduri, combinând un \code{MatTableDataSource} Angular Material pentru filtrare și sortare cu o proprietate calculată care aplică segmentarea pe datele filtrate. Această abordare permite paginarea corectă a grilei de carduri, unde soluția standard \code{MatPaginator} ar funcționa doar cu tabelele \code{mat-table}.

\section{Sistemul de programări}

\subsection{Ciclul de viață al unei programări}

O programare în MediLink trece prin mai multe stări definite ca \textit{enum} PostgreSQL: \code{pending} (creată de pacient, în așteptarea confirmării), \code{confirmed} (confirmată de medic sau asistent), \code{completed} (consultația a avut loc) și \code{cancelled} (anulată de oricare parte).

Tranziția dintre stări este controlată strict pe server: pacienții pot anula programările proprii, dar nu le pot confirma; asistenții și medicii pot confirma sau anula; marcarea ca \code{completed} se face exclusiv de medic sau asistent.

Modelul este instruit să ofere informații medicale generale bazate pe dovezi, să nu formuleze diagnostice definitive și să recomande consultul medical pentru situații urgente. \section{Mesageria în timp real}

\subsection{Arhitectura WebSocket}

Conexiunea WebSocket este menținută pe toată durata sesiunii utilizatorului, reutilizând același canal pentru notificări, mesagerie și semnalizare WebRTC. La conectare, serverul înregistrează conexiunea asociată ID-ului utilizatorului într-un dicționar în memorie.

\subsection{Serviciul Angular de WebSocket}

Pe frontend, conexiunea WebSocket este gestionată de un serviciu Angular singleton (\code{WebSocketService}), care:
\begin{itemize}
  \item Menține o singură conexiune pe sesiune, reconectând automat la deconectare;

  \item Gestionează autentificarea conexiunii prin trimiterea token-ului JWT la conectare.
\end{itemize}

\section{Exportul GDPR}

La cererea pacientului, platforma generează un document HTML complet cu toate datele personale și medicale stocate. Documentul este structurat pe secțiuni: date personale, profil medical (alergii, boli cronice, grup sanguin), istoricul programărilor, fișele medicale complete (cu datele decriptate), semnele vitale, prescripțiile și mesajele. Exportul este disponibil și în format JSON structurat, pentru interoperabilitate cu alte sisteme.

Funcționalitatea de export implementează articolul 20 din GDPR privind portabilitatea datelor. Generarea documentului este asincronă: la apăsarea butonului de export, se afișează un indicator de progres, iar documentul HTML este livrat ca fișier descărcabil după finalizarea generării pe server. Toate datele criptate în baza de date (diagnostice, tratamente, rezultate analize) sunt decriptate automat înainte de includerea în export, astfel încât utilizatorul primește datele în format lizibil.

\section{Configurarea și deploymentul cu Docker}

\subsection{Structura serviciilor Docker}

Stack-ul MediLink este definit în \code{docker-compose.yml} și cuprinde cinci servicii interconectate: \code{db} (PostgreSQL 16), \code{redis} (Redis 7), \code{backend} (FastAPI), \code{frontend} (Angular compilat) și \code{nginx} (reverse proxy). Fiecare serviciu are propriul Dockerfile sau imagine oficială, variabilele de mediu injectate din fișierul \code{.env} și volumele necesare pentru persistența datelor.

\subsection{Configurarea Nginx}

Nginx acționează ca punct unic de intrare pentru tot traficul platformei, definindu-se trei blocuri de rutare distincte: cererile cu prefix \code{/api} sunt transmise prin proxy către serviciul backend FastAPI cu header-ele de request originale, conexiunile WebSocket (\code{/ws}) sunt gestionate corect prin transmiterea header-elor \code{Upgrade} și \code{Connection}, iar toate celelalte cereri sunt servite din directorul cu fișierele Angular compilate. Directiva \code{try\_files} este esențială pentru rutarea Angular: la accesarea directă a unui URL profund (\code{/dashboard/vitals}), Nginx servește \code{index.html} și lasă Angular Router să gestioneze navigarea pe client.

\subsection{Variabile de mediu și securitate}

Toate secretele aplicației (cheia JWT, cheia de criptare Fernet, parola bazei de date, API key-ul Groq) sunt gestionate exclusiv prin variabile de mediu definite într-un fișier \code{.env} care nu este inclus în sistemul de versionare. Fișierul \code{.env.example} inclus în repository documentează toate variabilele necesare fără a dezvălui valorile reale.

\section{Arhitectura frontend Angular}

\subsection{Structura modulară a proiectului}

Proiectul Angular MediLink este organizat pe principiul \textit{feature-based architecture}: fiecare funcționalitate majoră este grupată într-un folder dedicat care conține componentele, serviciile și modelele aferente. Structura principală a proiectului este:

\begin{itemize}
  \item \code{/core} --- servicii singleton (ApiService, AuthService, WebSocketService), interceptori HTTP, garduri de rută;
  \item \code{/shared} --- componente reutilizabile (header, sidebar, dialogs), pipe-uri, directive;
  \item \code{/features/patient} --- toate componentele specifice rolului de pacient (dashboard, vitale, programări, chat AI);
  \item \code{/features/doctor} --- componentele medicului (lista pacienți, fișe medicale, predicție risc, raport PDF);
  \item \code{/features/assistant} --- gestionarea programărilor și atribuirea pacienților;
  \item \code{/features/admin} --- dashboard administrativ, gestionare utilizatori, jurnal audit.
\end{itemize}

\subsection{Serviciul API central}

Toată comunicarea cu backend-ul se realizează prin \code{ApiService}, un serviciu Angular singleton injectat prin dependency injection în toate componentele care au nevoie de date de la server. Acesta centralizează logica de construire a URL-urilor și expune metode tipizate pentru fiecare resursă a API-ului, returnând \code{Observable<T>} pentru fiecare apel. Tipizarea strictă TypeScript garantează că datele primite de la server sunt tratate corect în componentele consumatoare.

\subsection{Interceptorul HTTP pentru autentificare}

Un \textbf{interceptor HTTP} Angular interceptează automat fiecare cerere ieșantă și adaugă header-ul de autentificare JWT (\code{Authorization: Bearer <token>}), eliminând nevoia de a gestiona manual token-ul în fiecare serviciu. Interceptorul este înregistrat o singură dată la nivel de aplicație și se aplică transparent tuturor apelurilor HTTP, inclusiv celor din librăriile terțe care utilizează \code{HttpClient}.

\subsection{Garduri de rută}

Protecția rutelor Angular se realizează prin \textbf{route guards}, funcții executate înainte de navigarea la o rută, care pot bloca accesul sau redirecționa utilizatorul. MediLink implementează două garduri:

\begin{itemize}
  \item \textbf{AuthGuard} --- verifică dacă utilizatorul este autentificat; dacă nu, redirecționează la \code{/login};
  \item \textbf{RoleGuard} --- verifică dacă rolul utilizatorului curent corespunde rolului cerut de rută; dacă nu, redirecționează la \code{/unauthorized}.
\end{itemize}

Interfața MediLink este construită pe componentele Angular Material cu tema Material Design 3. \section{Gestionarea erorilor și logging}

\subsection{Tratarea erorilor pe backend}

MediLink implementează un sistem unificat de tratare a erorilor prin \textbf{exception handlers} globali FastAPI înregistrați la nivel de aplicație. Excepțiile sunt clasificate pe tipuri (HTTPException pentru erorile de business, Exception generală pentru erorile neașteptate) și returnează răspunsuri JSON consistente cu câmpurile \code{error}, \code{message} și \code{status\_code}. Erorile interne (500) sunt înregistrate în log cu stack trace complet, dar răspunsul către client nu expune detalii tehnice sensibile.

\subsection{Jurnalul de audit}

Acțiunile critice (autentificare reușită/eșuată, modificarea datelor medicale, exportul GDPR, generarea raportului AI) sunt înregistrate automat în tabelul \code{audit\_logs}. Fiecare înregistrare conține: ID-ul utilizatorului, acțiunea efectuată, resursa afectată, adresa IP, user agent-ul browserului și timestamp-ul.

Jurnalul de audit este accesibil administratorilor prin interfața dedicată, cu filtrare după utilizator, acțiune și interval de timp. Această funcționalitate permite investigarea incidentelor de securitate și demonstrarea conformității GDPR.

\subsection{Tratarea erorilor pe frontend}

Pe frontend, erorile HTTP sunt interceptate global de \textbf{ErrorInterceptor}, care afișează notificări \textit{snackbar} Material cu mesajul de eroare returnat de backend. Erorile de rețea (server indisponibil, timeout) afișează un mesaj generic și oferă opțiunea de reîncercare. Stările de loading sunt gestionate prin flag-uri booleene per componentă, cu afișarea de \textit{skeleton loaders} Material pentru o experiență de utilizare fluidă.

\section{Optimizări de performanță}

\subsection{Optimizări backend}

Performanța backend-ului MediLink a fost îmbunătățită prin mai multe tehnici:

\begin{itemize}
  \item \textbf{Lazy loading pentru relații SQLAlchemy} --- relațiile între entități sunt încărcate la cerere (\code{lazy='select'}), evitând interogările SQL costisitoare pentru datele neutilizate;
  \item \textbf{Indexuri PostgreSQL} --- câmpurile utilizate frecvent în filtrare (\code{user\_id}, \code{patient\_id}, \code{doctor\_id}, \code{appointment\_date}) au indecși B-tree dedicați;
  \item \textbf{Paginare server-side} --- listele lungi (fișe medicale, programări, notificări) sunt paginate pe server prin parametrii \code{skip} și \code{limit}, evitând transferul de date inutile;
  \item \textbf{Conexiuni pooled} --- SQLAlchemy gestionează un pool de conexiuni reutilizabile la baza de date, eliminând overhead-ul de reconectare per cerere.
\end{itemize}

\subsection{Optimizări frontend}

Angular oferă mai multe mecanisme native de optimizare utilizate în MediLink:

\begin{itemize}
  \item \textbf{Lazy loading} pentru modulele de rute --- fiecare secțiune majoră (doctor, pacient, admin) este încărcată la prima accesare, nu la pornirea aplicației; aceasta reduce semnificativ timpul de inițializare;
  \item \textbf{OnPush change detection} --- componentele cu date statice sau actualizate rar utilizează strategia \code{ChangeDetectionStrategy.OnPush}, reducând numărul de cicluri de detectare a schimbărilor;
  \item \textbf{TrackBy în liste} --- directiva \code{@for} cu \code{track} evită re-renderizarea completă a listelor la actualizarea parțială a datelor;
  \item \textbf{Compresie Gzip prin Nginx} --- bundle-ul Angular comprimat are dimensiuni reduse cu circa 70\% față de varianta necomprimată.
\end{itemize}

\section{Testarea aplicației}

\subsection{Strategia de testare}

Testarea MediLink urmează principiul \textbf{piramidei de testare}: o bază largă de teste unitare rapide, un strat intermediar de teste de integrare și, la vârf, un număr redus de teste end-to-end. Această distribuție maximizează acoperirea codului menținând o suită de teste rapidă și ușor de întreținut.

\subsection{Teste unitare și de integrare backend}

Suita de teste backend este construită cu \textbf{pytest} \cite{pytest2024} și include 60+ teste care acoperă:

\begin{itemize}
  \item \textbf{Autentificare} --- login cu credențiale valide/invalide, fluxul 2FA, refresh token, logout și invalidarea token-urilor;
  \item \textbf{CRUD resurse} --- crearea, citirea, actualizarea și ștergerea pentru toate entitățile principale (utilizatori, pacienți, medici, programări, fișe medicale, semne vitale);
  \item \textbf{Validare Pydantic} --- testarea schemelor cu date invalide și verificarea mesajelor de eroare returnate;
  \item \textbf{Control acces RBAC} --- verificarea că fiecare endpoint returnează 403 pentru roluri nepermise;
  \item \textbf{Criptare/decriptare} --- testarea \code{EncryptedString} cu diverse valori, inclusiv cazuri limită (șir gol, caractere speciale, valori null).
\end{itemize}

Baza de date utilizată în teste este o instanță SQLite în memorie, izolată de producție, cu rate limiting dezactivat prin variabila de mediu \code{TESTING=1}.

\subsection{Configurarea pytest}

Suita de teste este configurată prin fișierul \code{pytest.ini}, care specifică directorul de teste, modul asincron automat pentru funcțiile \code{async} și variabilele de mediu specifice testării. Baza de date utilizată este SQLite în memorie, izolată complet de baza de date de producție. Fiecare test primește un client HTTP proaspăt și o bază de date curată prin intermediul fixture-urilor pytest, garantând izolarea completă între teste. Rate limiting-ul este dezactivat în mediul de test prin variabila de mediu \code{TESTING=1}.

\subsection{Acoperirea codului}

Raportul de acoperire (\textit{code coverage}) generat prin \code{pytest-cov} indică o acoperire de \textbf{78\%} a codului backend, cu acoperire completă (100\%) pentru modulele critice de securitate: autentificare, generarea/validarea token-urilor, criptarea datelor și middleware-ul RBAC. Modulele cu acoperire mai redusă sunt cele care implică apeluri externe (API Groq, generare PDF) și care sunt excluse din testele automate prin mock-uri.

\section{Sistemul de notificări în timp real}

\subsection{Arhitectura notificărilor}

Sistemul de notificări al MediLink este construit în jurul unei conexiuni WebSocket persistente, menținute pe întreaga durată a sesiunii utilizatorului. La conectare, serverul înregistrează conexiunea în dicționarul \code{active\_connections}, indexat după ID-ul utilizatorului, permițând trimiterea de mesaje direcționate oricând pe parcursul sesiunii.

Notificările sunt structurate ca obiecte JSON cu câmpurile: \code{type} (tipul notificării), \code{title}, \code{message}, \code{link} (URL opțional pentru navigare directă) și \code{timestamp}. Pe frontend, serviciul \code{WebSocketService} distribuie mesajele primite prin intermediul unui \code{Subject} RxJS, la care se abonează componentele interesate.

\subsection{Tipurile de notificări generate}

MediLink generează notificări automate pentru mai multe evenimente de sistem:

\begin{itemize}
  \item \textbf{Confirmare programare} --- pacientul primește o notificare instantanee când asistentul sau medicul îi confirmă programarea, cu data și ora confirmată;
  \item \textbf{Anulare programare} --- notificare cu motivul anulării, dacă este furnizat;
  \item \textbf{Reminder programare} --- notificări automate la 24 de ore și 1 oră înainte de programare, generate de scheduler-ul APScheduler \cite{apscheduler2024};
  \item \textbf{Mesaj nou} --- notificare la primirea unui mesaj nou în conversație, cu previzualizarea primelor 50 de caractere;
  \item \textbf{Valoare vitală anormală} --- alertă generată automat la înregistrarea unui semn vital în afara intervalului de referință (ex: tensiune arterială sistolică > 140 mmHg);
  \item \textbf{Raport PDF generat} --- notificare că raportul medical solicitat este gata de descărcare.
\end{itemize}

\section{Modulul de mesagerie în timp real}

\subsection{Structura conversațiilor}

Mesageria în MediLink este organizată ca un sistem de conversații directe (1-la-1) între utilizatori. Fiecare conversație este identificată printr-o pereche ordonată de ID-uri de utilizatori, stocată în tabelul \code{messages}. La accesarea paginii de mesagerie, frontend-ul interoghează lista de conversații active ale utilizatorului curent, afișând ultimul mesaj și numărul de mesaje necitite pentru fiecare.

\subsection{Trimiterea și recepționarea mesajelor}

Trimiterea unui mesaj urmează un flux dublu pentru fiabilitate:

\begin{enumerate}
  \item Mesajul este mai întâi \textbf{persistat în baza de date} prin POST la \code{/api/messages}, garantând că nu se pierde în cazul unei deconectări WebSocket;
  \item \textbf{Simultan}, serverul trimite mesajul în timp real prin WebSocket destinatarului, dacă acesta este conectat;
  \item Dacă destinatarul nu este conectat în momentul trimiterii, mesajul va fi livrat la reconectare, prin interogarea bazei de date.
\end{enumerate}

Această arhitectură hibridă (REST pentru persistență + WebSocket pentru livrare în timp real) garantează că niciun mesaj nu se pierde, indiferent de starea conexiunii destinatarului.

\subsection{Indicatori de stare}

Interfața de mesagerie afișează indicatori vizuali pentru: mesajele necitite (bold + badge numeric), ora trimiterii fiecărui mesaj (formatată relativ: „acum 5 minute", „ieri la 14:30") și starea de online/offline a interlocutorului, determinată pe baza prezenței conexiunii WebSocket active.

\section{Generarea raportului medical PDF}

\subsection{Fluxul de generare}

Raportul medical PDF este una dintre funcționalitățile distinctive ale MediLink, combinând inteligența artificială cu generarea documentelor profesionale. Fluxul de generare parcurge mai mulți pași:

\begin{enumerate}
  \item Medicul apasă butonul „Generează raport PDF" din fișa unui pacient;
  \item Frontend-ul trimite o cerere POST la \code{/api/ai/report/{patient\_id}};
  \item Backend-ul colectează toate datele pacientului: profil medical, ultimele 20 de înregistrări medicale, semnele vitale din ultimele 90 de zile și prescripțiile active;
  \item Datele sunt trimise modelului Llama 3.3 70B prin API-ul Groq, cu un prompt structurat care solicită generarea unui raport narativ clinic complet;
  \item Textul generat de AI este integrat într-un template HTML structurat pe secțiuni medicale standard;
  \item \textbf{WeasyPrint} \cite{weasyprint2024} convertește documentul HTML+CSS în format PDF, cu antet și subsol pe fiecare pagină, numerotare pagini și stilizare profesională;
  \item PDF-ul generat este returnat clientului ca fișier descărcabil cu header-ul \code{Content-Disposition: attachment}.
\end{enumerate}

\subsection{Structura raportului generat}

Raportul medical PDF generat de MediLink este structurat în secțiuni clinice standard:

\begin{itemize}
  \item \textbf{Date de identificare} --- numele complet, data nașterii, CNP (mascat pentru confidențialitate), grupul sanguin;
  \item \textbf{Sumar clinic} --- rezumatul stării de sănătate generat de AI pe baza istoricului complet;
  \item \textbf{Diagnostice active și boli cronice} --- lista condițiilor medicale cunoscute cu data primei înregistrări;
  \item \textbf{Evoluția semnelor vitale} --- tabel cu valorile înregistrate în ultimele 90 de zile și tendința (crescătoare/descrescătoare/stabilă);
  \item \textbf{Medicația curentă} --- lista prescripțiilor active cu doze și instrucțiuni;
  \item \textbf{Istoricul consultațiilor} --- ultimele 10 consultații cu diagnostic și tratament;
  \item \textbf{Recomandări și plan de urmărire} --- recomandările generate de AI pe baza profilului de risc.
\end{itemize}
